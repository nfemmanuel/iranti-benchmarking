## Objective

Design and execute Benchmark B3: a test of whether Iranti's Librarian correctly resolves factual conflicts and maintains KB accuracy under adversarial write conditions.

This tests the **conflict resolution** subsystem documented in `docs/features/conflict-resolution/spec.md`.

---

## Benchmark Design

### Task Definition

A known ground truth is established for N entities × M facts. Then adversarial writes inject incorrect values with varying confidence deltas. We measure whether the final KB state reflects the ground truth.

### Arms

**Baseline arm:**
No conflict resolution system. Last write wins (or first write wins depending on implementation). A wrong value written after a correct value overwrites it with 100% probability (absent conflict detection).

**Iranti arm:**
Conflict detection fires when an incoming write conflicts with an existing value. Resolution uses weighted confidence: `weighted = raw × 0.7 + raw × reliability × 0.3`. If confidence gap ≥ 10, deterministic resolution. If gap < 10, LLM reasoning.

### Test Conditions

| Condition | Description | Expected Iranti Behavior |
|-----------|-------------|--------------------------|
| C1: High-confidence correct | Correct fact (conf 90) written first, then wrong fact (conf 70) | Correct wins (gap=20 → deterministic) |
| C2: High-confidence wrong | Wrong fact (conf 90) written first, then correct fact (conf 70) | Wrong wins (gap=20 → deterministic); this is a known limitation |
| C3: Close confidence | Correct (conf 78) then wrong (conf 75) | LLM reasoning triggered; expected: correct wins |
| C4: Very close confidence | Correct (conf 76) then wrong (conf 75) | LLM reasoning or escalation; ambiguous |
| C5: Reliability advantage | Correct from trusted source (reliability 0.8, conf 70) vs wrong from neutral source (reliability 0.5, conf 80) | Correct wins via reliability weighting |

### Metrics

- Resolution accuracy: proportion of conflict outcomes where the ground truth value is in the KB
- Escalation rate: proportion of conflicts escalated to human review
- LLM reasoning accuracy: when LLM is called, how often does it pick the correct value?

### Expected Results

Condition C1: Iranti correctly keeps the high-confidence correct fact (100% accuracy, deterministic).
Condition C2: Iranti incorrectly overwrites correct fact with high-confidence wrong fact. This is expected behavior, not a bug — it reflects the design decision that confidence is the primary signal. This should be documented as a known limitation.
Condition C3: LLM reasoning expected to favor correct value. Testing whether the LLM reasoning is better than chance.
Condition C4: Escalation likely. Tests the escalation pathway.
Condition C5: Tests whether source reliability learning changes outcomes.

---

## Threats to Validity

1. **Confidence as proxy**: In real systems, agents assign confidence scores. In this test, we set confidence programmatically. The mapping from "correct fact" to "high confidence" is assumed, not derived.
2. **LLM reasoning circularity**: If we use Claude to both set up and evaluate the benchmark, there may be coherence between the LLM's reasoning tendencies and its setup choices.
3. **Write arm dependency**: Cannot execute until writes work.
4. **Reliability baseline**: At test start, all sources have default reliability (0.5). Results reflect initial-state behavior, not steady-state learned reliability.
